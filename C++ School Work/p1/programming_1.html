4 Program description

4.1 Introduction

A team of biologists at UC Davis are planning to carry out a big research study, to better understand the impact of climate change and its effect on wildlife during forest fires. They have already chosen a special location, at the center of a forest, and will set up hundreds of sensors all around this center to collect data. The sensors will be activated when they detect animal movement and will record the distance from such event to the center. After the experiment, their goal is to be able to search for specific distances in their datasets.

Each events will be recorded as a position vector (). The distance of an event is computed as the vector’s magnitude: .

The problem is that they don’t yet know how many position vectors they will collect and how many magnitudes they will want to search and analyse.

  • On one hand, if the amount of position vectors is big, but they only need to search a few magnitudes within it, then using a linear search with a list of unordered vectors might be enough.

  • On the other hand, if they end up having to search a significant amount of magnitudes, then paying the one time cost to sort the list of vectors and using a binary search might be worth it.

This team of biologists is at the top of their field but none of them has taken ECS 36C so they don’t know much about computational complexity. That is why they are asking you to help figure out this problem and submit your educated opinion!

4.2 Goal #1: program

Write a program named vector_search.cc, that receives a file containing a list of unordered positional vectors and a file containing a list of magnitudes. According to the user’s choice, the program performs either a linear search or a binary search in order to find the given magnitudes within the list of vectors. It finally prints the search performance on the screen and saves the number of magnitudes that were found in an output file.

Example:

$ cat vectors_1.dat     # Print list of unordered positional vectors
4 4
8 4
-8 -6
5 3
-5 1
-2 2
10 1
-1 5
-6 -3
7 2
$ cat magnitudes_1.dat  # Print list of magnitudes
1
5
17
12
15
$ ./vector_search vectors_1.dat magnitudes_1.dat res_1.dat  # Run program
Choice of search method ([l]inear, [b]inary)?
l
CPU time: 10.777 microseconds
$ cat res_1.dat         # Print number of matches found (one match found: (4,4) => 5)
1
$

A working example of the program can be found at /home/cs36c/public/p1/vector_search_reference.

4.3 Goal #2: measurement analysis

By running the program multiple times and with inputs of different sizes (see dataset generator explained below), explain the pros and cons of linear search versus sorting first and using binary search, with respect to computational complexity. Try to explain if it is possible to determine a ratio between the number of vectors and the number of magnitudes to search, that makes either linear search or binary search the best option.

5.2 Phase 1: reading in the dataset

In this phase, your program should receive two filenames as command line arguments, the vector file and magnitude file, which the program should read in. The vectors and magnitudes should be stored in two std::vector containers.

Because some code to open these two files will probably be equivalent, you are encourage to write generic functions in order to avoid code duplication.

5.2.1 Vector file

The first file contains a list of positional vectors, one per line. Each line has two integer values, separated by a space. The first value is the x value, the second value is the y value.

Example of vector file:

$ cat vectors_1.dat
5 7
-2 1
...

You should write a class for representing positional vector, so that each vector is an object of that class. For debugging purposes, you can overload the << operator to print positional vectors in a pretty way.

5.2.2 Magnitude file

The second file contains a list of magnitudes, one per line and expressed as an integer value.

Example of magnitude file:

$ cat magnitudes_1.dat
1
5
17
...

5.3 Phase 2: searching strategies

In this phase, you will need to implement two search strategies in order to find the given magnitudes in the list of positional vectors: a linear search and a binary search.

The user will choose between using one of these two strategies every time the program is run. If the user input is invalid, they should be asked again, as shown in the example below. The error message should be printed to std::cerr.

$ ./vector_search vectors_1.dat magnitudes_1.dat res_1.dat
Choice of search method ([l]inear, [b]inary)?
J
Incorrect choice
2
Incorrect choice
l
CPU time: 10.777 microseconds
$

The general search algorithm is:

count = 0
for (m in magnitudes)
    if (search if m is in vectors)
        count += 1

For the binary search, the list of positional vectors should be sorted by their magnitude beforehand. For the sorting, you should use std::sort. If you overload the < operator of your positional vector class, this function can easily be applied for sorting all of objects contained in a vector container:

std::sort(vect_list.begin(), vect_list.end());

If two positional vector’s magnitudes are equal, you can order them by their y values.

Note that computing a positional vector’s magnitude will yield a floating-point value, while the magnitudes read for the magnitude file are expressed as integer values. When comparing these values, you can simply cast the floating-point values into simple integers.

5.4 Phase 3: processing time measurements

Create a class for counting the number of elapsed microseconds, which you will use to measure the performance of the search strategies. Your class should a high resolution clock, as defined by std::chrono::high_resolution_clock, and convert the time into microseconds for display.

Because the linter provided by Google doesn’t like the use of chrono, we have to turn the linter down for the include line specifically as shown in the example.

Here is an example of how to use the relevant functions defined in std::chrono:

#include <chrono> // NOLINT (build/c++11)
...
std::chrono::high_resolution_clock::time_point start;
start = std::chrono::high_resolution_clock::now();
... /* some computation happening */
auto end = std::chrono::high_resolution_clock::now();
double elapsed_us = std::chrono::duration<double, std::micro>(end - start).count();
std::cout << "CPU time: " << elapsed_us << " microseconds";
...

If you declare ct as an object of such class, your program’s main fragment of code could look like:

print "Choice of search method ([l]inear, [b]inary)?"
get choice from user
ct.Reset()
switch (choice)
    case 'l':
        run linear search
        break;
    case 'b':
        sort list of vectors
        run binary search
        break;
    default:
        print "Incorrect choice"
        go back to getting user choice
print "CPU time: " << ct.CurrentTime() << " ticks"

Optionally, you can overload the class’ << operator and simply display object ct directly (e.g.: std::cout << ct << std::endl;).
