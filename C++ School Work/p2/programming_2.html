3 Program #1: Postfix expression evaluation

3.1 Introduction

My mum, who is now happily retired, used to be a nuclear physicist. During most of her career, she used one of these ancient calculators made by HP in the 1980s, which were well known for their use of Revere Polish Notation, also known as postfix notation. It used to drive me crazy because it is a real brain effort to write postfix expressions instead of the standard infix expressions we are all used to.

For example, the infix expression 3 + 4 would be written 3 4 + in postfix notation. More complicated, the infix expression ((15 / (7 - (1 + 1))) * 3) - (2 + (1 + 1)) would be written 15 7 1 1 + - / 3 * 2 1 1 + + -. In the second example, we understand one of the main advantages of postfix expressions: they do not need parentheses! The order of evaluation is always explicit.

Interestingly enough, it is much easier to solve postfix expressions from a programming point of view, by using a single value stack.

3.2 Goal

Write a program named postfix_eval.cc that reads postfix expressions from the standard input and evaluates them, until there is no more expressions to read.

A postfix expression is a line of input composed of numbers (double) and operators (+, -, *, /).

Any errors detected in the expression should be reported on the screen. Error do not stop the program, incorrect expressions are reported as invalid and are simply ignored.

$ ./postfix_eval
1 3 +
4
2 3 1 * + 9 -
-4
3.14 2 *
6.28
1 3 /
0.333333
Bye!
$ ./postfix_eval
3 4
Error: invalid expression
4 +
Error: invalid expression
3 456 $
Error: unknown symbol '$'
12 34 45 *
Error: invalid expression
Bye!
$ echo $?
0
$

A working example of the program can be found at /home/cs36c/public/p2/postfix_eval_reference.

4 Program #2: Air luggage management

4.1 Introduction

Sacramento International Airport (SMF) is trying to better understand the process of luggage loading and unloading into airplanes to make it more efficient. The first step towards this goal is to create a computational model of the process.

Before departure, luggage is loaded into containers as the luggage from passengers arrives. When a container is full, or there is no more luggage, it is loaded into the airplane.

After arriving at destination, the containers are unloaded with the last on being the first off. As each container is unloaded, the bags it contains are unloaded in the same order they were loaded, i.e., last on being last off.

4.2 Goal

Write a program named luggage_handling.cc that receives two command line arguments, the name of an input file and a number. The file contains a list of bags (each represented by a unique integer), in the order that they arrive. The number on the command line (a positive integer) indicates the number of bag each container can hold.

Your program should output to the screen the order of the bags as they are unloaded, as shown in the example below.

$ cat bags25.dat
16 24 25 3 20 18 7 17 4 15 13 22 2 12 10 5 8 1 11 21 19 6 23 9 14
$ ./luggage_handling bags25.dat 7
6 23 9 14 10 5 8 1 11 21 19 17 4 15 13 22 2 12 16 24 25 3 20 18 7 
$ ./luggage_handling bags25.dat 10
19 6 23 9 14 13 22 2 12 10 5 8 1 11 21 16 24 25 3 20 18 7 17 4 15 
$
$ ./luggage_handling
Usage: ./luggage_handling <input_file> <container_size>
$ ./luggage_handling bags25.dat
Usage: ./luggage_handling <input_file> <container_size>
$ ./luggage_handling wrong_file 7
Error: cannot open file wrong_file
$ ./luggage_handling bags25.dat -1
Error: invalid container size
$ echo $?
1
$

A working example of the program can be found at /home/cs36c/public/p2/luggage_handling_reference.

4.3 Additional information and constraints

  • This program uses a clever use of stacks and queues. You can use pointers to queues and/or pointers to stacks, if it makes sense.
  • You cannot use std::stack or std::queue and have to provide a stack implementation with your code.
  • You can reuse the stack and queue implementations provided on Canvas, as part of the lecture material.
  • For this program, you should implement a bounded queue. A bounded queue receives a maximum capacity at its creation and cannot enqueue more than this capacity.
  • You are allowed to implement additional and relevant methods to the queue and stack implementations (e.g., IsEmpty() or IsFull()).
  • All error messages should be written to std::cerr and the program should exit with return code 1 upon error.
  • Some input samples are provided in /home/cs36c/public/p2/bags{25,37,100}.dat.
