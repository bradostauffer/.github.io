3 Phase #1: Indexed min-priority queue

3.1 Principle

An indexed priority queue is a special type of priority queues invented by Robert Sedgewick (he is the one behind Red-Black trees and the Quicksort algorithm) and Kevin Wayne.

Like regular priority queues, it is used to push items associated with priorities and pop them according to their priority (smallest value first for min-priority queues).

Indexed priority queues differ from regular priority queues is the sense that:

  • An item is composed of one comparable, generic key and one integer index value.
  • The key represents the priority of the item. The item with the minimum key will be at the top of a min-priority queue.
  • The value is necessarily an integer index and must belong to a pre-defined range, between 0 and capacity - 1.

Because of the nature of values (i.e., bounded integers), it is possible to represent all the key-value pairs via an array of size capacity. The array can be directly indexed by a value to find the corresponding key.

This interesting optimization enables to provide operation bool Contains(unsigned int idx) in (instead of for a regular priority queue), which by extension improves the complexity of operation Changekey(K key, unsigned int idx) too.

Additional resources:

  • Complete Java implementation: https://algs4.cs.princeton.edu/24pq/IndexMinPQ.java.html
  • Associated javadoc: https://algs4.cs.princeton.edu/code/javadoc/edu/princeton/cs/algs4/IndexMinPQ.html

3.2 Implementation strategy

A significant portion of the implementation is already provided to you in /home/cs36c/public/p5/index_min_pq.h. As you can notice, the implementation is largely based on a regular heap-based priority queue, as shown in lecture.

It differs in the way pairs of key-value are stored and organized. The IndexMinPQ class defines three vectors:

  • keys[]: array of keys, indexed by the value.
  • heap_to_idx[]: mapping between heap indexes (e.g., root node has index 1 in a heap) and index values.
  • idx_to_heap[]: inverse mapping between index values and heap indexes.
    • In other words: idx_to_heap[heap_to_idx[i]] == i.
    • An invalid mapping is represented by the value 0.

Let’s consider an example to illustrate how these vectors are used. Assuming the following code:

// Indexed min-priority queue of capacity 100
IndexMinPQ<double> impq(100);

// Insert a bunch of key-value
std::vector<std::pair<double, int>> keyval{
    { 2.2, 99},
    { 51.0, 54},
    { 42.5, 53},
    { 74.32, 93}
};
for (auto &i : keyval) {
    impq.Push(i.first, i.second);
}

4 Phase #2: Dijkstra’s shortest path algorithm

4.1 Introduction

Graphs are one of the most versatile data structures. They can represent any collection of interconnected entities, and are thus widely used to model real-life issues. For example, it is possible to model all the roads in the USA using a graph: the vertices are the intersection between roads, and edges are the segments of road themselves.

A fundamental graph algorithm is to find the shortest path between two vertices in a graph. This is exactly what your GPS does when giving you the directions between two locations.

With a naive approach, the shortest path would be the path that contains the least amount of edges between the source and destination vertices. But in many cases, it is actually common to associate each edge with a weight: e.g., you can often go faster on the highway, than on a secondary road. The shortest path then becomes the path that has the smallest cumulated weight, regardless of the number of edges.

One of the most popular algorithms to find the shortest path between two vertices was invented by Dijkstra, more than half of a century ago. One efficient implementation of it, which you will write, uses a min-priority queue.

4.2 Dijkstra’s shortest path algorithm

Assuming a graph G containing a collection of vertices connected by directed edges, the pseudo of Dijkstra’s algorithm that find the shortest path between vertice src and vertice dst is as follows.

function Dijkstra(G, src, dst):
    create min-priority queue Q

    for each vertex v in G:
        dist[v] = INFINITY                  # Unknown distance from src to v
        prev[v] = UNDEFINED                 # Predecessor of v in path

    dist[src] = 0                           # Distance from src to itself is 0

    Q.add_with_priority(src, dist[src])     # Add first vertex to queue

    while Q is not empty:                   # The main loop
        u = Q.extract_min()                 # Remove and return closest vertex
        if u == dst:                        # Reached destination
            break loop
        for each neighbor v of u:           # Go through all the neighbors
            alt = dist[u] + length(u, v)    # Compute alternate path that goes through u
            if alt < dist[v]:               # Alternate path is better than existing one
                dist[v] = alt               # Update distance from src to v
                prev[v] = u                 # Update predecessor of v in path
                if Q.Contains(v):           # Update priority queue
                    Q.change_priority(v, dist[v])
                else
                    Q.add_with_priority(v, dist[v])


    # Backtrack to find the shortest path between src and dst, if any
    S = empty sequence
    u = dst
    if dist[u] is not INFINITY:             # Do something only if the vertex is reachable
        while u is defined:                 # Construct the shortest path in S
            insert u at the beginning of S  # Insert the vertex into S
            u = prev[u]                     # Traverse from destination to source

    return S, dist[dst]                     # Shortest path from src to dst, and its total weight

4.4 Additional information and constraints

  • The nature of this problem can be nicely expressed with object-oriented programming. So do not use global variables, and think in terms of objects.

    For example, the graph can be an object made of a collection of vertices. The vertices can be instantiated from another class, and be described as a collection of edges. Finally, edges are yet another class, and are described by a destination and a weight.
    • As usual, you can overload the << operator of these classes in order to debug objects easily.

    In the same way, the shortest path can be expressed as an object itself. It is initialized with a source vertex and a destination vertex, and passed as an argument to the graph’s Dijkstra method, which completes it with the shortest path information that can later be displayed.

  • The Dijkstra’s shortest path algorithm needs a min-priority queue in order to keep track of vertices. Because vertices are sequentially numbered, using an indexed priority queue is very efficient. Edges can be added to the queue, using the destination vertex’s number as the index and the edge’s weight as the key. After Phase #1, you should have a working indexed min-priority queue that you can use in this phase.
